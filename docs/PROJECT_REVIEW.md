# 项目构建记录与总结

## 1. 项目定位

本项目以“懂车帝汽车数据”为主题，完整实现了一条面向大模型应用的数据工程链路，覆盖：

- 数据抓取
- 数据清洗与结构化
- 数据质量校验与报告
- 训练/RAG 数据产物输出
- 向量化与检索
- LLM 生成问答
- 检索重排
- 轻量存储层
- RAG 效果评估

项目目标不是只做一个“能抓网页的脚本”，而是尽可能贴近真实业务中的大模型数据开发流程，形成从采集到评估的闭环。

---

## 2. 最终项目结构

当前项目可以分为 6 个层次：

### 2.1 抓取层

- 主入口：[main.py](file:///e:/workplace/Crawl/main.py)
- 抓取实现：[dcd_scraper.py](file:///e:/workplace/Crawl/scraper/dcd_scraper.py)

职责：

- 采集懂车帝车系详情页数据
- 保存原始 HTML 与 JSON
- 支持失败重试、请求限速、车系池扩展

### 2.2 清洗与加工层

- 清洗模块：[dcd_cleaner.py](file:///e:/workplace/Crawl/cleaner/dcd_cleaner.py)
- 文本加工：[llm_processor.py](file:///e:/workplace/Crawl/utils/llm_processor.py)

职责：

- 从原始页面中提取结构化字段
- 转为适合大模型理解的 Markdown
- 按标题和语义进行文本切分
- 为下游训练和检索生成 chunk

### 2.3 数据质量与工程化层

- 质量规则：[data_quality.py](file:///e:/workplace/Crawl/quality/data_quality.py)
- 任务状态：[job_state.py](file:///e:/workplace/Crawl/utils/job_state.py)
- 统计采集：[stats_collector.py](file:///e:/workplace/Crawl/utils/stats_collector.py)
- Parquet 导出：[parquet_exporter.py](file:///e:/workplace/Crawl/utils/parquet_exporter.py)

职责：

- 清洗数据和训练样本质量校验
- 输出质量报告与可视化图表
- 支持断点续跑、失败重试、任务汇总
- 生成 JSONL/Parquet 等标准数据产物

### 2.4 检索与生成层

- 标准 RAG 主脚本：[rag_llm_demo.py](file:///e:/workplace/Crawl/rag_llm_demo.py)
- Embedding 客户端：[embedding_client.py](file:///e:/workplace/Crawl/utils/embedding_client.py)
- 向量存储：[faiss_vector_store.py](file:///e:/workplace/Crawl/utils/faiss_vector_store.py)
- LLM 客户端：[llm_client.py](file:///e:/workplace/Crawl/utils/llm_client.py)
- 重排器：[reranker_client.py](file:///e:/workplace/Crawl/utils/reranker_client.py)

职责：

- 构建向量库
- 稠密检索 + 稀疏检索融合
- 模型级 reranker 可切换接入
- 基于上下文约束的 RAG 回答生成

### 2.5 存储与追踪层

- 轻量存储层：[light_storage.py](file:///e:/workplace/Crawl/utils/light_storage.py)

职责：

- 将 chunk 元数据写入 DuckDB
- 记录 build 过程与 query 日志
- 为后续评测、分析、回溯提供基础存储

### 2.6 评估层

- 评估脚本：[rag_eval.py](file:///e:/workplace/Crawl/rag_eval.py)
- 评估文档：[RAG_EVAL.md](file:///e:/workplace/Crawl/docs/RAG_EVAL.md)

职责：

- 自动生成评测集
- 计算 HitRate@K、Recall@K、MRR
- 评估引用准确率和回答保守性

## 3. 版本迭代记录

下面按照项目的实际演进路径，总结各版本的能力变化。

### V0：原型期 - 抓取脚本可跑通

阶段目标：

- 找到真实可抓取站点
- 用 Python 跑通抓取链路
- 保留原始页面数据

实现结果：

- 放弃过于理想化的目标站点，切换到更适合做真实业务数据工程演示的懂车帝汽车数据
- 建立基础目录结构：`scraper / cleaner / utils / data / docs`
- 初步实现页面抓取与原始 HTML、JSON 落盘

这一阶段的意义：

- 证明“数据拿得到”
- 完成从空项目到真实站点接入的第一步

### V1：数据工程基础版 - 清洗、结构化、落盘

阶段目标：

- 把抓到的数据变成可用于大模型的高质量语料

实现结果：

- 提取车系基础信息、价格、评分、配置、资讯等字段
- 输出清洗后的 JSON 与 Markdown
- 生成训练数据 `JSONL`
- 引入元数据字段：`source`、`url`、`title`、`series_id`、`crawl_timestamp`、`chunk_index` 等

这一阶段的意义：

- 从“网页内容”升级为“结构化语料”
- 让数据具备进入训练与 RAG 的基础条件

### V2：数据质量与工程增强版

阶段目标：

- 让数据链路具备真实工程系统的最小能力

实现结果：

- 增加数据质量规则：空值、异常值、重复、乱码、字段完整性校验
- 输出质量报告：JSON、Markdown、字段覆盖率、分布图
- 支持 Parquet 输出
- 引入断点续跑、失败重试、限速、日志分级、抓取统计、任务汇总
- 扩大车系池，提升数据量到千条级训练样本

这一阶段的意义：

- 从“演示脚本”升级为“可连续运行的数据流水线”
- 项目开始具备大厂面试中常说的“工程性”

### V3：最小可用 RAG 版

阶段目标：

- 验证数据是否真的能服务下游

实现结果：

- 新增最小版 RAG 脚本：[rag_demo.py](file:///e:/workplace/Crawl/rag_demo.py)
- 使用轻量哈希向量实现离线检索
- 在无 API 条件下完成最小问答闭环

这一阶段的意义：

- 证明“数据链路能跑通下游应用”
- 适合作为教学与理解向量检索原理的过渡版本

### V4：标准 RAG 版

阶段目标：

- 从玩具 Demo 升级到接近真实业务的标准 RAG 架构

实现结果：

- 引入真实 Embedding：
  - `ollama`
  - `openai_compatible`
  - `sentence_transformers`
  - `fastembed`
- 引入 FAISS 向量库
- 引入 LLM 生成能力
- 加入“仅基于上下文回答”的提示词约束
- 输出带来源引用的回答

这一阶段的意义：

- 完成“数据工程 -> 检索系统 -> 大模型回答”的标准闭环
- 技术栈已经进入真实项目的主流范式

### V5：检索增强版

阶段目标：

- 解决“向量分数高但答案不稳定”的问题

实现结果：

- 将检索升级为 `hybrid`：
  - FAISS 稠密检索
  - BM25 稀疏召回
  - RRF 融合重排
- 引入关键词覆盖特征，降低答非所问概率
- 支持调试输出重排明细

这一阶段的意义：

- 从“单塔向量检索”升级到“混合检索”
- 更贴近大厂实际线上方案

### V6：模型级重排与降级增强版

阶段目标：

- 接入真正的 reranker，同时保证链路鲁棒性

实现结果：

- 支持可切换 `Cross-Encoder` 重排
- 引入 `fail-open` 自动降级策略
- 当 `torch`/DLL 环境异常时，自动回退到轻量重排而非整个流程中断

这一阶段的意义：

- 展现“效果优先 + 可用性兜底”的工程设计思想
- 体现真实生产环境中的容错意识

### V7：存储与评估闭环版

阶段目标：

- 让系统具备可追踪、可复盘、可评估能力

实现结果：

- 新增 DuckDB 轻量存储层，落地：
  - `chunks`
  - `build_runs`
  - `query_logs`
- 新增 RAG 效果评估工具：
  - 自动生成问题集
  - 检索指标评估
  - 引用准确率评估
  - 回答保守性评估

这一阶段的意义：

- 项目从“能跑”升级到“能衡量、能分析、能迭代”
- 已具备真实项目复盘和持续优化的基础

## 4. 项目过程中遇到的关键问题与修复

### 4.1 数据字段缺失问题

现象：

- 部分页面中 `user_info` 为 `null`，导致清洗时报错

处理方式：

- 增加空值保护逻辑，避免因个别字段异常导致整条数据失败

工程意义：

- 提高爬虫对脏数据的容忍度

### 4.2 变量使用顺序错误

现象：

- `markdown_text` 在某些路径下先使用后赋值，导致 `UnboundLocalError`

处理方式：

- 调整生成顺序，确保落盘前文本内容已经准备好

工程意义：

- 说明单次跑通不等于稳定，必须经过流程级验证

### 4.3 Ollama Embedding 端点兼容问题

现象：

- 不同版本 Ollama 对 `/api/embed` 和 `/api/embeddings` 的兼容性不同

处理方式：

- 在 [embedding_client.py](file:///e:/workplace/Crawl/utils/embedding_client.py) 中增加双端点自动尝试逻辑

工程意义：

- 提升对环境差异的兼容性

### 4.4 本地依赖 DLL 初始化失败

现象：

- `torch` 或 `onnxruntime` 在虚拟环境中出现 DLL 初始化失败

处理方式：

- 对 Embedding 和 Reranker 方案增加可替代路径
- 对 reranker 引入自动降级逻辑

工程意义：

- 这是典型的“生产环境依赖不稳定”问题
- 项目通过降级策略体现了系统鲁棒性

### 4.5 检索排序与最终回答不一致

现象：

- 检索结果高分项与最终回答涉及车系不一致

处理方式：

- 引入混合检索与重排
- 强化上下文约束提示词
- 提供调试输出，便于分析排序与引用关系

工程意义：

- 说明 RAG 的核心问题不是“接入一个模型”，而是“检索质量与生成一致性”

## 7. 当前项目与大厂真实项目的主要差距

虽然项目已经很完整，但如果对标大厂生产系统，还存在明显差距。

### 7.1 数据规模仍偏小

当前特点：

- 千条级别样本适合学习和 demo

大厂真实情况：

- 通常是百万到亿级文档
- 数据来源远不止一个站点

优化方向：

- 扩站点采集
- 支持增量更新
- 引入更稳定的数据调度机制

### 7.2 存储层仍偏轻

当前特点：

- DuckDB 适合单机分析和轻量回溯

大厂真实情况：

- 结构化元数据通常进入 Hive、ClickHouse、MySQL、Postgres 或数据湖
- 向量通常落到 Milvus、Elastic、PGVector、Vearch、Qdrant 等

优化方向：

- 将 DuckDB 升级为“本地开发模式”
- 增加可切换的生产级存储后端

### 7.3 检索链路仍是单机版

当前特点：

- FAISS 本地文件索引适合中小规模场景

大厂真实情况：

- 往往有在线索引服务、分片、冷热分层、索引增量更新

优化方向：

- 增加在线索引更新能力
- 增加检索服务 API 化
- 设计统一召回接口

### 7.4 评估仍偏轻量

当前特点：

- 已有自动评测集和基础指标

大厂真实情况：

- 通常会区分离线评估、在线评估、A/B Test、人工标注集、业务指标联动

优化方向：

- 引入人工标注集
- 增加问题类型分桶评估
- 引入业务维度指标，例如点击、停留、满意度

### 7.5 编排与部署能力不足

当前特点：

- 以脚本驱动为主

大厂真实情况：

- 任务调度常用 Airflow、Argo、Dagster、内部平台
- 服务部署有容器化、监控告警、灰度发布

优化方向：

- 增加 Docker 化
- 增加调度脚本或工作流编排
- 增加监控和告警埋点

### 7.6 数据治理仍不够系统

当前特点：

- 已经有数据质量规则，但还不是严格的数据治理体系

大厂真实情况：

- 会有 schema 管理、版本管理、血缘追踪、权限治理、脱敏审计

优化方向：

- 明确 schema version
- 给训练数据和索引数据增加 version 字段
- 建立数据血缘记录：raw -> cleaned -> training -> index -> eval

## 8. 对标大厂真实项目的下一步优化路线

如果继续把这个项目往“大厂真实项目”方向升级，建议按下面的优先级推进。

### P1：服务化与调度化

- 把抓取、构建索引、评估拆为可独立触发的任务
- 提供统一 CLI + 配置文件
- 再进一步可封装为 HTTP API 或工作流

### P2：生产级存储与向量库

- 元数据切到 Postgres / ClickHouse
- 向量库切到 Milvus / PGVector / Qdrant
- 保留 DuckDB 作为本地开发与分析模式

### P3：增量索引与版本管理

- 支持新增/更新/删除文档的增量索引
- 增加数据版本、索引版本、评测版本
- 支持版本回滚与结果对比

### P4：更强评测体系

- 构建人工高质量问题集
- 区分检索类问题、比较类问题、价格类问题、推荐类问题
- 加入 LLM-as-a-Judge 与人工复核结合的评估方式

### P5：线上可观测性

- 增加接口耗时、召回命中、回答失败率、引用覆盖率监控
- 增加日志聚合和告警

### P6：多路召回与特征工程

- 增加基于标题、品牌、价位、车型标签的结构化召回
- 增加 query 改写、意图识别、实体归一化
- 增加 session 级别的多轮检索优化

### P7：数据治理与安全

- 增加采集规则说明
- 增加敏感字段检查
- 增加数据使用边界、版本记录和审计留痕
