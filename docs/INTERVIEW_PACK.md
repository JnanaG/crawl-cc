# 简历项目描述 + 面试问答清单

## 1. 简历项目描述

### 项目名称

汽车领域大模型数据工程与标准 RAG 系统

### 项目简介

基于懂车帝真实汽车页面数据，独立完成从数据抓取、清洗、结构化、质量评估，到向量化检索、混合召回、重排、LLM 生成与效果评测的完整链路，构建了一套面向汽车知识问答场景的标准 RAG 系统。

### 简历版描述（推荐 3-4 条）

- 独立搭建汽车领域大模型数据工程项目，基于懂车帝真实页面实现 `raw -> cleaned -> processed -> vector_store -> eval` 全链路数据分层，产出 `JSONL`、`Parquet`、DuckDB 等多种下游格式。
- 实现数据采集与加工流水线，支持失败重试、请求限速、断点续跑、日志分级、抓取统计、质量报告与任务汇总，构建可持续迭代的数据工程基础设施。
- 搭建标准 RAG 系统，接入真实 Embedding、FAISS 向量库、`Hybrid Retrieval`（稠密检索 + BM25 + RRF）、可切换 `Cross-Encoder` 重排与 LLM 生成，提升检索相关性与回答稳定性。
- 实现轻量评测闭环，支持自动生成评测集并统计 `HitRate@K`、`Recall@K`、`MRR`、引用准确率等指标，形成“数据构建 -> 检索问答 -> 评估优化”的工程闭环。

### 简历版描述（更偏技术细节）

- 基于 Python 构建汽车数据采集系统，完成懂车帝页面抓取、字段提取、Markdown 清洗、语义分块与训练数据生成，支撑千条级 RAG 语料构建。
- 设计数据质量规则引擎，对清洗记录和训练样本执行空值、异常值、重复、乱码、字段完整性校验，并输出 JSON/Markdown 质量报告及分布图。
- 使用 Ollama/OpenAI 兼容 Embedding + FAISS 构建向量检索链路，并引入 BM25 稀疏召回、RRF 融合、关键词覆盖特征及 Cross-Encoder reranker 优化召回排序效果。
- 新增 DuckDB 轻量存储层记录 chunk 元数据、构建日志与查询日志，结合自动评测脚本实现 RAG 结果可追踪、可评估、可复盘。

### 一句话亮点版

- 从真实站点抓取开始，独立实现了一个具备工程化能力、评测闭环和混合检索能力的汽车领域标准 RAG 项目。

我做了一个汽车领域的大模型数据工程与标准 RAG 项目，数据源来自懂车帝真实页面。整个项目不是只做爬虫或只做问答，而是完整打通了数据抓取、清洗、质量校验、结构化输出、向量化检索、LLM 生成和效果评估的闭环。

在数据侧，我设计了 `raw / cleaned / processed / vector_store / eval` 分层，支持 JSONL、Parquet 和 DuckDB 输出；在工程侧，加入了失败重试、断点续跑、日志分级、任务统计和质量报告；在 RAG 侧，使用了真实 Embedding、FAISS、BM25、RRF 融合和可切换 Cross-Encoder 重排，提高检索相关性和回答稳定性；最后还补了评测脚本，能统计 HitRate@K、MRR 和引用准确率。

这个项目让我比较完整地实践了大模型数据工程和 RAG 系统的核心链路，也让我理解了大厂项目里为什么强调“数据质量、检索效果和评估闭环”。

### 版本 B：更偏工程化

我这个项目的核心目标，不是做一个能跑的问答 Demo，而是做一套接近真实业务的 RAG 数据基础设施。前半段是数据工程，包括懂车帝页面抓取、结构化清洗、训练语料构建、质量规则和 Parquet 输出；后半段是标准 RAG，包括真实 Embedding、FAISS、混合检索、重排、LLM 生成和效果评估。

我在里面重点补了几类工程能力：第一是稳定性，比如重试、限速、断点续跑；第二是可观测性，比如日志、统计、质量报告和检索日志；第三是效果优化，比如 hybrid retrieval 和 Cross-Encoder reranker；第四是评估闭环，比如自动生成评测集和计算 HitRate@K、MRR、引用准确率。整体上，它更像一个“可持续迭代的 RAG 系统雏形”，而不是单次演示脚本。

## 4. 高频面试问答清单

### Q1：你为什么做这个项目？

参考回答：

我想做的不只是一个爬虫项目，而是一套完整的大模型数据工程项目。因为在真实业务里，RAG 的效果很大程度上不是由模型本身决定的，而是由数据质量、检索策略和评估机制决定的。所以我选了一个真实站点做数据源，完整实现了从数据抓取到 RAG 问答再到效果评测的闭环。

### Q2：这个项目的核心价值是什么？

参考回答：

核心价值有两个。第一，它验证了真实业务数据可以被清洗、结构化并接入标准 RAG 链路；第二，它不是“只会回答”，而是具备可追踪、可评估、可优化的工程闭环，这一点更接近实际业务项目。

### Q3：为什么选择懂车帝？

参考回答：

因为汽车领域数据天然适合做 RAG。它既有结构化字段，比如价格、车型、车身尺寸、动力参数，也有半结构化文本，比如口碑、资讯、描述文案。这样的数据非常适合做数据清洗、切分、检索和问答，也方便我展示数据工程和 RAG 两部分能力。

### Q4：你的数据链路是怎么设计的？

参考回答：

我采用了 `raw / cleaned / processed / vector_store / eval` 分层。`raw` 保存原始 HTML 和 JSON，便于回溯；`cleaned` 保存结构化清洗结果和 Markdown；`processed` 保存训练/RAG 用 JSONL 和 Parquet；`vector_store` 保存向量索引和记录；`eval` 保存评测集、评测报告和明细结果。这样做的好处是数据链路清晰，便于复用和定位问题。

### Q5：你做了哪些数据质量工作？

参考回答：

我对清洗数据和训练样本都做了质量校验，包括空值过滤、异常值检测、乱码识别、重复数据检测和字段完整性检查。然后输出 JSON 和 Markdown 格式的质量报告，并补充字段覆盖率和分布图。这样做的目的是让数据质量可见，而不是只靠人工看几条样本。

### Q6：为什么还要输出 Parquet？

参考回答：

因为 JSONL 更适合训练和逐条处理，但 Parquet 更适合分析和批量处理。很多真实数据工程链路都会同时保留面向训练的格式和面向分析的格式。Parquet 在列式存储、压缩和后续分析方面更有优势，也更接近真实数仓或数据湖场景。

### Q7：你为什么需要断点续跑和失败重试？

参考回答：

爬虫和数据处理本身就是高失败概率链路，如果每次中断都从头跑，成本很高，也不符合真实工程要求。所以我增加了任务状态持久化、失败重试、跳过已完成任务和任务统计，这样流程中断后能继续跑，也更容易分析失败原因。

### Q8：你为什么先做了一个轻量版 RAG？

参考回答：

因为我想先快速验证“数据链路是否能服务下游问答”，所以先做了一个最小可用版本，用轻量哈希向量在无 API 条件下跑通检索逻辑。之后再升级成真实 Embedding + FAISS + LLM 的标准 RAG。这种方式能降低前期依赖，适合逐步演进。

### Q9：你的标准 RAG 架构是什么？

参考回答：

我的标准 RAG 链路是：训练语料切分后做 Embedding，向量写入 FAISS；查询时先做召回，再将 Top-K 上下文送给 LLM 生成答案，并在 prompt 中要求仅基于上下文回答并保留引用。Embedding 支持 Ollama、本地模型和 OpenAI 兼容接口，生成模型支持 Ollama 和 OpenAI 兼容接口。

### Q10：为什么从 dense retrieval 升级到 hybrid retrieval？

参考回答：

因为我实际观察到一个问题，就是有些问题在 dense retrieval 下虽然相似度高，但召回结果不够稳定，容易出现“向量相关但语义不完全对题”的情况。于是我加了 BM25 稀疏召回，并用 RRF 做融合，再加上关键词覆盖特征做二阶段打分。这样对实体词、品牌词、车型词更敏感，实际更贴近业务需求。

### Q11：为什么还要加 reranker？

参考回答：

混合检索解决的是“召回更全”，但不一定能保证最终排序最优。reranker 的作用是对候选集做更细粒度的语义判断，所以我又增加了可切换的 Cross-Encoder 重排。这样能进一步提升排序质量，尤其对比较复杂的问题更有帮助。

### Q12：如果 Cross-Encoder 加载失败怎么办？

参考回答：

我做了 fail-open 自动降级策略。如果 `torch` 或 DLL 环境异常，系统不会直接报错退出，而是回退到轻量重排链路继续跑。因为在真实工程里，效果增强是加分项，但链路可用性更重要，所以要优先保证系统可运行。

### Q13：为什么你没有直接用 LangChain 搭全套？

参考回答：

我刻意没有把核心流程全部交给 LangChain 封装，是因为我想更清楚地理解底层机制，包括数据切分、向量构建、召回、融合、重排、引用约束和评估逻辑。LangChain 在业务里可以作为编排层，但如果不知道底层原理，出了效果问题很难定位。这个项目更强调对核心链路的掌控力。

### Q14：你的项目里，最难的问题是什么？

参考回答：

我觉得最难的不是“代码写出来”，而是处理真实链路中的不稳定性和效果偏差。比如有的页面字段缺失、不同环境下 `torch` 或 `onnxruntime` 出现 DLL 问题、Ollama 接口版本差异、检索排序和最终回答不一致等。每个问题背后都不是单纯改一行代码，而是要考虑兼容性、降级策略和可观测性。

### Q15：你是怎么做评估的？

参考回答：

我补了一个 `rag_eval.py` 脚本，先根据当前语料自动生成评测集，然后计算 HitRate@K、Recall@K、MRR；如果开启生成评估，还会统计 citation_precision、citation_hit_rate、回答是否带引用、是否倾向于“无法确认”等指标。这样做的目的是把效果优化从“主观感觉”变成“可量化迭代”。

### Q16：你认为这个项目和大厂真实项目的差距在哪里？

参考回答：

差距主要在 4 个方面：第一是数据规模，我现在还是千条级；第二是存储和检索基础设施，目前还是单机版 DuckDB + FAISS；第三是评测体系，当前还以轻量离线评估为主；第四是服务化和调度能力，目前更多是脚本化运行。如果继续做，我会优先往生产级向量库、增量索引、评测平台和在线服务化方向升级。

### Q17：如果让你继续优化，你会先做什么？

参考回答：

我会优先做三件事。第一，补高质量人工评测集，细分价格类、对比类、推荐类问题；第二，把检索与索引能力服务化，支持增量更新；第三，把存储层升级成可切换的生产后端，比如 Postgres/ClickHouse + Milvus/PGVector。这样项目会更接近真实线上系统。

### Q18：这个项目里你最有成就感的部分是什么？

参考回答：

我最有成就感的是把项目从“抓几页网页”逐步演进成“有数据分层、有质量规则、有混合检索、有重排、有存储、有评估”的完整系统。因为这个过程让我真正理解了，大模型项目不是只接 API，而是一个完整的数据与系统工程问题。

## 5. 面试官可能追问的问题

### 追问 1：为什么不用 Elasticsearch？

你可以答：

- 这个项目当前重点是把数据工程和标准 RAG 主链路跑通
- 所以我先选择了更轻量、上手更快的 FAISS
- 如果进入生产，我会优先考虑 ES 做稀疏检索、Milvus/PGVector 做向量检索，或者直接用支持混合检索的统一检索引擎

### 追问 2：为什么 DuckDB 只算轻量存储层？

你可以答：

- DuckDB 很适合本地分析和开发态回溯
- 但它不是典型的在线高并发服务存储
- 所以更适合作为单机分析工具，而不是线上主存储

### 追问 3：为什么你说 RAG 的关键是数据和检索，而不只是模型？

你可以答：

- 因为如果召回错了，模型再强也只能在错误上下文里生成
- 实际项目中很多问题不是模型不会答，而是没拿到正确证据
- 所以数据清洗质量、chunk 策略、召回质量、重排质量，往往比单纯换一个更大的模型更关键

### 追问 4：你这个项目如果在线上，会怎么部署？

你可以答：

- 抓取和离线构建部分用调度系统定时跑
- 索引构建和评估做成离线任务
- 检索和问答封装成在线服务
- 增加监控、日志聚合、失败告警和版本管理

## 7. 最终建议

如果你把这个项目用于简历和面试，建议准备三套说法：

- 30 秒版本：一句话讲目标和亮点
- 2 分钟版本：完整讲架构、技术方案和工程化能力
- 5 分钟版本：展开讲版本迭代、问题修复、评估闭环和未来优化

你真正需要让面试官感受到的，不是“你写了很多代码”，而是：

- 你能把一个模糊目标拆成完整工程链路
- 你能处理真实环境中的问题
- 你能用数据、检索、评估思维做大模型项目

这才是这个项目最有价值的部分。
