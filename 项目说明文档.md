## 1. 项目简介

### 1.1 项目背景

本项目是一个面向大模型应用的**汽车领域数据工程项目**，从懂车帝网站采集汽车车系数据，经过清洗、结构化处理后，生成适合大模型训练和RAG（检索增强生成）应用的高质量数据集。

### 1.2 项目目标

- **数据采集**：自动化抓取懂车帝车系详情页，包括车型配置、价格、评分、新闻等多维度信息
- **数据清洗**：将原始HTML/JSON转换为结构化的三层数据架构（Raw → Cleaned → Training）
- **质量保障**：建立完整的数据质量检查规则和评估报告体系
- **RAG应用**：提供从简单到生产级的多种RAG实现方案
- **工程能力**：支持断点续跑、失败重试、限速控制、日志追踪等生产级特性

### 1.3 技术栈

**核心依赖**：
- `requests` - HTTP请求
- `beautifulsoup4` - HTML解析
- `pandas` - 数据处理
- `tiktoken` - Token计数（OpenAI分词器）
- `loguru` - 日志管理
- `pyarrow` - Parquet格式支持

**RAG相关**：
- `faiss-cpu` - 向量检索
- `sentence-transformers` / `fastembed` - 文本向量化
- `duckdb` - 轻量级数据库存储
- `langchain` / `langchain-core` / `langchain-community` - LangChain编排与FAISS集成

**可视化**：
- `matplotlib` - 数据分布图表

### 1.4 项目特色

1. **完整的数据工程流水线**：从采集到应用的全链路实现
2. **三层数据架构**：清晰的数据分层，便于追溯和复用
3. **生产级工程能力**：断点续跑、重试机制、质量报告
4. **多种RAG实现**：从最简单的哈希向量到生产级的FAISS+LLM
5. **LangChain扩展**：新增LangChain版RAG实现，便于串联检索、Prompt和生成链路
6. **可评估可优化**：提供完整的RAG效果评估框架

### 2.2 数据流转

1. **采集阶段**：`DongchediScraper` 从懂车帝网站抓取车系页面，保存HTML和JSON到 `data/raw/`
2. **清洗阶段**：`DongchediCleaner` 提取结构化字段，生成JSON和Markdown到 `data/cleaned/`
3. **处理阶段**：`LLMDataProcessor` 进行文本分块，`DataQualityEngine` 执行质量检查
4. **导出阶段**：`ParquetExporter` 导出JSONL和Parquet格式到 `data/processed/`
5. **应用阶段**：通过RAG脚本构建向量库，执行检索和生成

## 3. 核心模块说明

### 3.1 爬虫模块 (scraper/)

#### DongchediScraper

**功能**：负责从懂车帝网站抓取车系数据

**核心方法**：
- `collect_series_ids(target_count, max_expand_requests)` - 收集车系ID池
  - 从首页获取种子ID
  - 通过关联页面扩展ID池
  - 支持限制扩展请求次数
  
- `fetch_series_data(series_id)` - 抓取单个车系详情
  - 获取HTML页面
  - 提取`__NEXT_DATA__` JSON数据
  - 保存原始HTML和JSON到`data/raw/`
  - 返回解析后的JSON和抓取元数据

**工程特性**：
- 请求节流：`min_interval_sec` 控制请求间隔
- 失败重试：支持指数退避重试机制
- User-Agent伪装：模拟浏览器请求
- 超时控制：防止请求挂起

### 3.2 清洗模块 (cleaner/)

#### DongchediCleaner

**功能**：将原始JSON转换为结构化的清洗数据

**核心方法**：
- `extract_clean_series_record(raw_json)` - 提取结构化字段
  - 解析车系基础信息（品牌、车型、城市）
  - 提取价格信息（经销商价、指导价、成交价）
  - 提取评分信息（综合评分、各维度评分）
  - 提取车型列表（配置、价格、标签）
  - 提取尺寸、图片、新闻等信息
  - 返回符合schema的结构化数据

- `clean_record_to_markdown(clean_record)` - 生成Markdown文本
  - 将结构化数据转换为LLM友好的Markdown格式
  - 保留层级结构和关键信息
  - 适合文本分块和向量化

**数据字段**：
- `series` - 车系基础信息
- `pricing` - 价格信息
- `scores` - 评分信息
- `dimensions` - 尺寸信息
- `images` - 图片/颜色信息
- `models` - 车型配置列表
- `news` - 新闻资讯列表
- `stats` - 统计摘要

### 3.3 质量模块 (quality/)

#### DataQualityEngine

**功能**：执行数据质量检查和生成质量报告

**核心方法**：
- `validate_clean_record(record)` - 校验清洗层数据
  - 检查必填字段完整性
  - 检测异常值和乱码
  - 检查重复数据
  - 返回校验结果和问题列表

- `validate_training_item(item)` - 校验训练层数据
  - 检查元数据完整性
  - 检查文本长度和Token数
  - 检测乱码文本
  - 返回是否通过和错误信息

- `generate_quality_report(...)` - 生成质量报告
  - 统计有效/无效样本数量
  - 计算Token分布统计
  - 计算字段覆盖率
  - 生成分布图表
  - 输出JSON和Markdown报告

**质量规则**：
- 强校验：不通过则标记为invalid或过滤
- 弱校验：通过但记录warning
- 详见 `docs/QUALITY_RULES.md`

### 3.4 工具模块 (utils/)

#### LLMDataProcessor

**功能**：文本分块和Token计数

**核心方法**：
- `count_tokens(text)` - 计算文本Token数
  - 使用tiktoken分词器（cl100k_base）
  - 兼容OpenAI GPT系列模型

- `chunk_text(text, max_chunk_tokens, overlap_tokens)` - 智能文本分块
  - 按Markdown标题拆分section
  - 按段落和句子切分语义单元
  - 在Token限制内合并单元
  - 超长单元使用窗口切分兜底
  - 支持重叠窗口

**分块策略**：
1. 标题级切分：保持章节完整性
2. 语义级切分：保持段落和句子完整性
3. Token级切分：超长内容的兜底策略

#### ParquetExporter

**功能**：导出Parquet格式数据

**核心方法**：
- `export_training_items(training_items, output_path)` - 导出训练数据
- `export_clean_summary(clean_records, output_path)` - 导出清洗摘要

**优势**：
- 列式存储，查询效率高
- 压缩率高，节省存储空间
- 支持复杂数据类型
- 与大数据生态兼容

#### JobStateStore

**功能**：管理任务状态，支持断点续跑

**核心方法**：
- `can_process(series_id, max_retry)` - 判断是否可处理
- `mark_running(series_id)` - 标记任务运行中
- `mark_success(series_id, meta)` - 标记任务成功
- `mark_failed(series_id, error, meta)` - 标记任务失败
- `mark_skipped(series_id, reason)` - 标记任务跳过

**状态文件**：`data/state/job_state.json`

**状态类型**：
- `pending` - 待处理
- `running` - 运行中
- `success` - 成功
- `failed` - 失败
- `skipped` - 跳过

#### StatsCollector

**功能**：收集任务统计信息

**核心方法**：
- `add_success(duration_sec, retries, http_status)` - 记录成功任务
- `add_failed(duration_sec, retries, http_status, error)` - 记录失败任务
- `add_skipped()` - 记录跳过任务
- `export(output_dir)` - 导出统计报告

**统计指标**：
- 成功/失败/跳过数量
- 重试次数统计
- 耗时统计（avg/median/p95/max）
- HTTP状态码分布
- 错误样本列表

#### EmbeddingClient

**功能**：文本向量化客户端

**支持的Provider**：
- `openai_compatible` - OpenAI兼容API
- `ollama` - Ollama本地服务
- `fastembed` - 本地Python模型

**核心方法**：
- `embed_texts(texts)` - 批量向量化
- `embed_query(query)` - 单条查询向量化

#### FaissVectorStore

**功能**：FAISS向量检索库

**核心方法**：
- `build(vectors, records)` - 构建索引
- `save(index_path, records_path, meta_path)` - 保存索引
- `load(index_path, records_path, meta_path)` - 加载索引
- `search(query_vector, top_k)` - 向量检索

**索引类型**：使用FAISS的IndexFlatIP（内积检索）

#### LLMClient

**功能**：LLM生成客户端

**支持的Provider**：
- `openai_compatible` - OpenAI兼容API
- `ollama` - Ollama本地服务

**核心方法**：
- `chat(system_prompt, user_prompt, temperature)` - 对话生成

#### RerankerClient

**功能**：重排序客户端

**支持的Provider**：
- `cross_encoder` - 交叉编码器模型（如bge-reranker）

**核心方法**：
- `score(query, texts)` - 计算相关性分数

#### LightRAGStorage

**功能**：轻量级DuckDB存储

**核心方法**：
- `refresh_chunks(records)` - 刷新chunk记录
- `log_build_run(...)` - 记录构建日志
- `log_query(...)` - 记录查询日志

**用途**：
- 记录向量库构建历史
- 记录查询历史和命中结果
- 便于评估和回溯

## 4. 数据架构

### 4.1 三层数据模型

本项目采用**三层数据架构**，清晰分离原始数据、清洗数据和训练数据：

```
Raw Layer (原始层)
    ↓
Cleaned Layer (清洗层)
    ↓
Training Layer (训练层)
```

### 4.2 Raw Layer (原始层)

**目录**：`data/raw/dongchedi/`

**文件格式**：
- `series_<series_id>.html` - 原始HTML页面
- `series_<series_id>.json` - 提取的`__NEXT_DATA__` JSON

**用途**：
- 保留页面原始数据，便于回放和调试
- 支持选择新的提取字段而无需重新抓取
- 作为数据溯源的最终依据

**特点**：
- 不可逆的源数据，不做任何裁剪
- HTML和JSON一一对应
- 文件名包含series_id作为主键

### 4.4 Training Layer (训练层)

**目录**：`data/processed/`

**文件格式**：
- `dongchedi_training_data.jsonl` - JSONL格式
- `dongchedi_training_data.parquet` - Parquet格式

**JSONL Schema**：

```json
{
  "metadata": {
    "source": "dongchedi",
    "url": "车系URL",
    "title": "车系名称",
    "series_id": "车系ID",
    "brand_name": "品牌名称",
    "car_type": "车型类别",
    "model_count": "车型数量",
    "news_count": "新闻数量",
    "crawl_timestamp": "抓取时间戳",
    "chunk_index": "分块索引",
    "total_chunks": "总分块数",
    "tokens": "Token数量"
  },
  "text": "分块后的文本内容"
}
```

**特点**：
- 每条记录是一个chunk（文本片段）
- 包含完整的元数据，便于追溯和过滤
- Token数量已预计算，便于向量化和生成
- 支持JSONL（流式处理）和Parquet（分析查询）两种格式

**用途**：
- 直接用于向量化和RAG检索
- 用于大模型微调训练
- 用于评估和分析

### 4.5 数据演进规范

**版本管理**：
- `schema_version` 从 `v1` 开始
- 字段变更必须升级版本号
- 保持向后兼容或提供迁移脚本

**字段扩展**：
- 新增字段优先进入Cleaned Layer
- 评估后决定是否进入Training Layer
- 删除字段前检查下游依赖

**命名规范**：
- 使用snake_case命名
- 字段名清晰表达含义
- 避免缩写和歧义

## 6. RAG应用

本项目提供三种RAG实现方案，并配套一个评估工具，从最小可用验证到 LangChain 编排逐步递进。

### 6.1 简单RAG演示 (rag_demo.py)

**特点**：
- 使用哈希向量（无需外部API）
- 离线运行，快速验证
- 适合学习和原型验证

**构建向量库**：

```powershell
python rag_demo.py build
```

**执行查询**：

```powershell
python rag_demo.py query --question "预算30万左右，推荐哪些SUV？"
```

**输出**：
- 基于检索结果的规则化回答
- 参考来源列表（车系名称、URL、相似度分数）

**限制**：
- 哈希向量精度较低
- 无LLM生成，仅拼接检索结果
- 不适合生产环境

### 6.2 标准RAG演示 (rag_llm_demo.py)

**特点**：
- 真实Embedding模型（OpenAI/Ollama/FastEmbed）
- FAISS向量检索
- LLM生成回答
- 支持混合检索（Dense + Sparse + RRF融合）
- 可选重排序（Cross-Encoder）
- DuckDB存储查询历史

**构建向量库**：

```powershell
# 使用OpenAI兼容API

$env:OPENAI_API_KEY = "your_key"
python rag_llm_demo.py build --embedding-provider openai_compatible --embedding-model text-embedding-3-small

# 使用Ollama
$env:OLLAMA_BASE_URL = "http://localhost:11434"
python rag_llm_demo.py build --embedding-provider ollama --embedding-model nomic-embed-text

# 使用本地模型
python rag_llm_demo.py build --embedding-provider fastembed --embedding-model BAAI/bge-small-zh-v1.5

# 使用sentence-transformers本地目录
python rag_llm_demo.py build `
  --embedding-provider sentence_transformers `
  --embedding-model D:\models\bge-small-zh-v1.5 `
  --batch-size 128
```

**执行查询**：

```powershell
# 基础查询
python rag_llm_demo.py query `
  --question "保时捷718适合什么人群？" `
  --embedding-provider ollama `
  --embedding-model nomic-embed-text `
  --llm-provider ollama `
  --llm-model qwen2.5:7b

# 混合检索 + 重排序
python rag_llm_demo.py query `

  --reranker-model D:\models\bge-reranker-base `

# 查看送入模型的上下文
```

**检索模式**：
- `dense` - 仅FAISS稠密向量检索
- `hybrid` - 稠密+稀疏+RRF融合（推荐）

**输出**：
- LLM生成的回答（基于检索上下文）

### 6.4 RAG效果评估 (rag_eval.py)

**特点**：
- 自动生成评测集
- 检索指标：HitRate@K、Recall@K、MRR
- 生成指标：引用准确率、回答长度
- 支持批量评估

**生成评测集**：

```powershell
python rag_eval.py prepare `
  --records data/vector_store/faiss/dongchedi_records.jsonl `
  --output data/eval/rag_eval_set.jsonl `
  --max-samples 80
```

**仅检索评估**：

```powershell
python rag_eval.py run `
  --eval-set data/eval/rag_eval_set.jsonl `
  --retrieval-mode hybrid `
  --embedding-provider ollama `
  --embedding-model nomic-embed-text `
  --top-k 6
```

**检索+生成评估**：

```powershell
python rag_eval.py run `
  --eval-set data/eval/rag_eval_set.jsonl `
  --retrieval-mode hybrid `
  --embedding-provider ollama `
  --embedding-model nomic-embed-text `
  --with-generation `
  --llm-provider ollama `
  --llm-model qwen2.5:7b
```

**评估指标**：
- `HitRate@K` - Top-K中至少命中一个正确结果的比例
- `Recall@K` - Top-K中正确结果的覆盖率
- `MRR` - 第一个正确结果的倒数排名均值
- `citation_precision` - 引用的准确率
- `citation_hit_rate` - 至少有一条正确引用的比例
- `avg_answer_length` - 平均回答长度

**输出文件**：
- `data/eval/rag_eval_report.json` - 汇总报告
- `data/eval/rag_eval_details.jsonl` - 明细结果

**建议阈值**（学习/求职项目）：
- `HitRate@6 >= 0.75`
- `MRR >= 0.45`
- `citation_precision >= 0.60`

## 7. 质量保障

### 7.1 质量检查规则

项目在Cleaned Layer和Training Layer分别执行质量检查，详见 `docs/QUALITY_RULES.md`。

**Cleaned Layer强校验**：
- `series.series_id` 不能为空
- `series.series_name` 不能为空
- `series.brand_name` 不能为空
- `series.car_type` 不能为空
- `pricing.dealer_price_range` 与 `pricing.official_price_range` 不能同时为空
- `stats.model_count` 必须大于0
- `series.series_name` 不能疑似乱码

**Cleaned Layer弱校验**：
- `scores.total_score` 超出常见区间（<0 或 >500）
- `models` 内存在重复 `car_id`
- `news` 中存在空标题

**Training Layer强校验**：
- `metadata` 必填字段不能缺失
- `text` 长度不能小于80
- `metadata.tokens` 必须大于0且为整数
- `text` 不能疑似乱码

**Training Layer弱校验**：
- `metadata.tokens < 30`（过小）
- `metadata.tokens > 1500`（过大）

### 7.2 质量报告

每次运行主流程后自动生成质量报告：

**JSON报告** (`data/reports/quality_report.json`)：

- 字段覆盖率（百分比）
- 无效样本明细

**Markdown报告** (`data/reports/quality_report.md`)：
- 人类可读的报告格式
- 包含分布图文件路径

**分布图**：

### 7.3 任务统计

- 成功/失败/跳过数量

## 8. 工程能力

### 8.1 断点续跑

**实现机制**：
- 使用 `JobStateStore` 管理任务状态
- 状态文件：`data/state/job_state.json`
- 每个车系记录状态：pending/running/success/failed/skipped

**状态转换**：
```
pending → running → success
                 → failed (retry < max_retry)
                 → failed (retry >= max_retry) → skipped
```

**使用场景**：
- 程序异常中断后继续运行
- 网络故障后恢复运行
- 增量更新数据

**历史数据复用**：
- 已成功的车系会跳过抓取
- 尝试从 `data/cleaned/` 回填训练样本
- 避免重复劳动，节省时间

### 8.2 失败重试

**重试策略**：
- 最大重试次数：3次（可配置）
- 指数退避：`2^(attempt-1)` 秒
- 随机抖动：避免雷鸣群效应

**重试触发条件**：
- HTTP 5xx错误
- 网络超时
- 连接异常

**重试记录**：
- 记录每次重试的错误信息
- 统计总重试次数
- 输出到任务汇总报告

### 8.3 限速控制

**请求节流**：
- 最小请求间隔：0.8秒（可配置）
- 随机抖动：0.05-0.2秒
- 防止触发反爬机制

**实现方式**：
- 记录上次请求时间戳
- 计算距离上次请求的时间差
- 不足最小间隔则sleep补足

### 8.4 日志系统

**日志级别**：
- DEBUG：详细的调试信息
- INFO：关键流程信息
- WARNING：警告信息（如跳过车系）
- ERROR：错误信息（如抓取失败）

**日志输出**：
- 控制台：INFO级别及以上
- 文件：DEBUG级别及以上
- 文件路径：`data/logs/run_YYYYMMDD_HHMMSS.log`

**日志内容**：
- 时间戳
- 日志级别
- 模块名称
- 日志消息

### 8.5 统计收集

**实时统计**：
- 成功/失败/跳过数量
- 重试次数
- 耗时分布
- HTTP状态码分布

**统计指标**：
- 平均耗时：`sum(duration) / count`
- 中位数耗时：`median(duration)`
- P95耗时：`percentile(duration, 0.95)`
- 最大耗时：`max(duration)`

**统计输出**：
- JSON格式：`data/reports/task_summary.json`
- Markdown格式：`data/reports/task_summary.md`

**错误记录**：

- 记录发生错误的车系ID

**容错设计**：
- 单个车系失败不影响整体流程

### B. 常见问题

**Q: 如何增加采集的车系数量？**
A: 设置环境变量 `TARGET_SERIES_POOL` 和 `TARGET_TRAINING_RECORDS`。

**Q: 如何调整文本分块的粒度？**
A: 设置环境变量 `MAX_CHUNK_TOKENS`，推荐范围100-500。

**Q: 如何查看任务执行状态？**
A: 查看 `data/state/job_state.json` 和 `data/reports/task_summary.json`。

**Q: 如何使用本地LLM进行RAG？**
A: 使用Ollama部署本地模型，然后使用 `rag_llm_demo.py` 的 `--llm-provider ollama` 参数。

**Q: 如何评估RAG效果？**
A: 使用 `rag_eval.py` 生成评测集并运行评估，查看HitRate、MRR等指标。
