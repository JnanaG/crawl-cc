# Crawl_cc 项目说明文档

## 1. 项目概述

`Crawl_cc` 是一个围绕懂车帝汽车车系数据构建的完整数据工程与 RAG 项目。项目不是单点爬虫，也不是单点问答 Demo，而是把以下几条链路打通成了一个相对完整的工程闭环：

- 数据采集
- 数据清洗与结构化
- 图片数据抽取与多模态资产构建
- 智能治理与质量守门
- 训练/RAG 语料分块与导出
- 向量索引构建与检索问答
- 离线评测与低分样本回流
- 工作流编排、告警与端到端验收
- 多轮对话、会话记忆与在线 API

从代码结构看，本项目的重点在于“以工程化方式组织大模型数据底座”，而不是单独优化某一个环节。

## 2. 项目目标

本项目试图解决三个核心问题：

### 2.1 如何把网页数据变成可用语料

项目从懂车帝车系页面抓取原始 HTML 与 `__NEXT_DATA__` JSON，再清洗成结构化 JSON 与适合 LLM/RAG 消费的 Markdown。

### 2.2 如何在进入下游前做质量治理

项目引入治理编排层，对样本做路由、规则去重、语义去重、质量打分和最终守门，避免低质量内容直接进入训练集和检索库。

### 2.3 如何形成可迭代闭环

项目不仅支持问答，还支持离线评测、结果回流、工作流状态管理、质量漂移告警和多轮对话记忆，具备持续优化基础。

## 3. 当前能力总览

### 3.1 数据生产能力

- 采集懂车帝车系详情页
- 保存原始 HTML 与核心 JSON
- 抓取阶段同步保存原始图片文件与图片 manifest
- 清洗为统一 schema 的结构化记录
- 转换为带章节结构的 Markdown
- 基于 token 进行 chunk 切分
- 导出 `JSONL + Parquet`
- 从 raw JSON 抽取车系图片资产，生成图片 manifest 与分类候选集
- 基于多模态 caption 将图片理解结果文本化，并并入现有 RAG 语料

### 3.2 智能治理能力

- 五节点治理主链路：`route -> rule_dedup -> semantic_dedup -> quality -> decision`
- 支持规则优先、启发式增强、LLM 可选增强
- 支持精确去重、规范化去重、语义去重
- 输出 `accept / repair / review / drop`
- 记录节点级审计日志、trace_id、batch_id

### 3.3 RAG 能力

- 支持最小离线 hash 检索
- 支持真实 embedding + FAISS 索引
- 支持 dense / hybrid 检索
- 支持轻量 BM25、RRF 融合
- 支持可选 reranker
- 支持基于上下文约束的 LLM 生成回答

### 3.4 评测与回流能力

- 支持传统检索评测
- 支持 `ragas` 优先、轻量 fallback 的多维评测
- 支持低分样本回流至 `feedback / review / repair`
- 支持 `ci_gate` 阈值守门

### 3.5 工作流能力

- 串联 `main -> build -> eval -> feedback -> ci_gate`
- 支持 `dry-run`
- 支持 `resume`
- 支持步级重试
- 支持工作流状态落盘
- 支持质量波动告警

### 3.6 多轮对话能力

- 支持会话创建、消息历史、偏好记忆、任务记忆
- 支持 query rewrite
- 支持推荐、比较、追问和澄清场景
- 支持 hash / faiss 双后端
- 支持 FastAPI 服务与最小前端页面

## 6. 主流程详解

主流程入口为 `main.py`。

### 6.1 初始化阶段

`main.py` 会初始化以下组件：

- `DongchediScraper`
- `DongchediCleaner`
- `LLMDataProcessor`
- `ParquetExporter`
- `DataQualityEngine`
- `JobStateStore`
- `StatsCollector`
- `GovernanceOrchestrator`

同时会从环境变量中读取：

- `TARGET_TRAINING_RECORDS`
- `TARGET_SERIES_POOL`
- `MAX_CHUNK_TOKENS`
- `MAX_SERIES_EXPAND_REQUESTS`
- `AGENT_ENABLE_LLM`
- `AGENT_LLM_PROVIDER`
- `AGENT_LLM_MODEL`
- `AGENT_LLM_API_BASE`
- `AGENT_LLM_API_KEY`
- `AGENT_LLM_FAIL_OPEN`

### 6.2 车系池收集

采集前，爬虫会通过多个入口收集车系 ID：

- 本地 raw 缓存
- 首页
- 公共车库/榜单页
- 搜索页关键词
- 车系详情页关联扩展

目的是在有限请求预算下尽可能扩充覆盖范围。

### 6.3 单条车系处理流程

对每个 `series_id`，主流程依次执行：

1. 检查是否允许处理，支持断点续跑和失败重试控制。
2. 抓取页面并保存原始 HTML、JSON。
3. 解析为 cleaned record。
4. 转换为 Markdown。
5. 执行 cleaned 层质量校验。
6. 进入治理编排器，得到 route、dedup、quality、decision。
7. 若治理决策为 `accept` 或 `repair`，则做 chunk 切分并验证每个训练项。
8. 保存 cleaned JSON、cleaned Markdown 和 processed 数据。
9. 更新状态、统计和审计。

### 6.4 产物输出

主流程输出的主要产物包括：

- `data/raw/dongchedi/`
- `data/cleaned/dongchedi/json/`
- `data/cleaned/dongchedi/markdown/`
- `data/processed/dongchedi_training_data.jsonl`
- `data/processed/dongchedi_training_data.parquet`
- `data/cleaned/dongchedi/cleaned_series.parquet`
- `data/reports/quality_report.json`
- `data/reports/quality_report.md`
- `data/reports/task_summary.json`

## 7. 智能治理链路详解

治理主入口是 `agent_pipeline/orchestrator.py` 中的 `GovernanceOrchestrator`。

### 7.1 治理主链路

```text
route -> rule_dedup -> semantic_dedup -> quality -> decision
```

### 7.2 route 节点

输入：

- `clean_record`

处理方式：

- 先用规则做基础路由
- 再用启发式逻辑做 refine
- 若启用了 LLM 且基础置信度不高，则再由 LLM 微调

输出：

- `channel`
- `route_decision`
- `template_version`
- `confidence`
- `reason`

### 7.3 rule_dedup 节点

处理方式：

- 计算内容 hash
- 计算规范化文本 hash
- 查询 dedup manifest

输出：

- 是否重复
- 重复类型
- 匹配记录
- 置信度

### 7.4 semantic_dedup 节点

处理方式：

- 从语义去重存储中找候选
- 若候选相似度超过阈值，则判断为语义重复
- 同车系与跨车系使用不同阈值

特点：

- 对低信息密度记录可以跳过深度判断
- 与规则去重协同工作

### 7.5 quality 节点

处理方式：

- 先用规则质量引擎打底
- 再用启发式规则做补充扣分
- 如启用 LLM，则在 base quality 上做受控增强

输出：

- `quality_score`
- `quality_tier`
- `issues`
- `issue_groups`
- `repair_suggestion`
- `rag_readiness`
- `training_readiness`

### 7.6 decision 节点

判定逻辑大致为：

- 重复样本直接 `drop`
- 路由置信度低转 `review`
- 质量分过低或 `rag_readiness` 太低转 `review`
- 中等质量样本转 `repair`
- 其余通过 `accept`

### 7.7 治理审计

治理层会记录：

- `trace_id`
- `batch_id`
- 每个节点的输入摘要
- 每个节点的输出摘要
- 节点耗时
- 是否使用了 LLM
- 最终治理结果

这使得治理层具备较强可解释性和可回溯性。

## 8. 数据资产分流

脚本入口：

- `scripts/split_processed_assets.py`

核心实现：

- `agent_pipeline/dataset_splitter.py`

分流后会得到三类资产：

### 8.1 `rag_corpus`

特点：

- 偏向检索可用性
- 关注 `rag_readiness`

### 8.2 `training_corpus`

特点：

- 偏向训练质量
- 对 `training_readiness` 和 `quality_score` 要求更高
- 会限制每个车系最多保留的 chunk 数

### 8.3 `eval_candidates`

特点：

- 基于车系维度聚合
- 自动推断可提问的事实类型
- 自动生成建议问题
- 作为黄金集扩写和评测候选池

## 9. RAG 系统设计

### 9.1 最小版 RAG

入口：

- `rag_demo.py`

特点：

- 使用本地 hash 向量
- 无需外部模型服务
- 适合快速验证问答链路

### 9.2 标准版 RAG

入口：

- `rag_llm_demo.py`

支持能力：

- 构建 FAISS 向量库
- 使用真实 embedding
- 支持 dense / hybrid 检索
- 支持 LLM 问答生成
- 支持 DuckDB 记录 build/query 日志

### 9.3 检索模式

`rag_llm_demo.py` 支持：

- `dense`
- `hybrid`

其中 `hybrid` 由以下几部分组成：

- FAISS 稠密检索
- 轻量 BM25 稀疏召回
- RRF 融合
- query term coverage 特征
- 可选 reranker 重排

### 9.4 Embedding 适配

`utils/embedding_client.py` 支持：

- `sentence_transformers`
- `fastembed`
- `openai_compatible`
- `ollama`

说明：

- `fastembed` 适合本地离线场景
- `openai_compatible` 和 `ollama` 适合接外部或本地模型服务
- `ollama` 已兼容不同 embedding API 端点

### 9.5 LLM 适配

`utils/llm_client.py` 支持：

- `openai_compatible`
- `ollama`

标准问答提示约束为：

- 必须仅基于上下文回答
- 上下文不足时明确说无法确认
- 尽量附带引用来源

### 9.6 存储层

`utils/light_storage.py` 使用 DuckDB 管理：

- `chunks`
- `build_runs`
- `query_logs`

用途：

- 回溯 chunk 元数据
- 记录索引构建任务
- 记录查询日志

## 10. 评测与回流设计

### 10.1 基础评测

入口：

- `rag_eval.py`

功能：

- 基于 records 自动生成简易评测集
- 计算命中率、召回率、MRR
- 可选生成阶段的引用相关指标

### 10.2 ragas 评测

入口：

- `rag_ragas_eval.py`

特点：

- 优先使用真实 `ragas`
- 如环境依赖有问题，会自动降级为 lightweight 评测
- 即使降级，也会继续输出标准结果文件

评测指标包括：

- `faithfulness`
- `answer_relevancy`
- `context_precision`
- `context_recall`

### 10.3 低分样本回流

入口：

- `scripts/sync_ragas_feedback.py`

核心实现：

- `agent_pipeline/feedback_bridge.py`

回流逻辑：

- 读取评测结果
- 识别低分 case
- 标记问题原因
- 分发到 `feedback / review / repair`

常见问题标签：

- `query_failure`
- `empty_context`
- `coverage_gap`
- `retrieval_noise`
- `grounding_weak`
- `answer_mismatch`

## 12. 多轮对话系统设计

多轮对话模块主要位于：

- `conversation/`
- `memory/`

### 12.1 核心流程

单轮对话执行顺序如下：

1. 创建或读取 `session`
2. 写入用户消息
3. 读取最近消息
4. 合并用户偏好记忆
5. 读取任务记忆
6. 识别本轮路由
7. 做 query rewrite
8. 生成 memory summary
9. 判断是否需要澄清
10. 调用 RAG 后端
11. 更新任务记忆
12. 写入助手回复

### 12.2 路由类型

当前支持：

- `fact_qa`
- `recommend`
- `compare`
- `follow_up`
- `clarify`

### 12.3 记忆设计

`memory/store.py` 基于 SQLite 维护：

- `sessions`
- `messages`
- `preferences`
- `task_memory`

其中：

- `preferences` 保存预算、能源类型、车型偏好、品牌偏好、关注点
- `task_memory` 保存任务类型、阶段、当前焦点车系、候选车系、上次改写 query

### 12.4 Query Rewrite

`conversation/query_rewriter.py` 当前能处理：

- 代词消解，如“它”“这款”“那款”
- 比较补全，如“那和银河 L7 比呢”
- 偏好补齐，如预算、能源、车型和关注点拼接

### 12.5 RAG 后端

`conversation/rag_backend.py` 支持：

- `hash` 后端
- `faiss` 后端

特点：

- `hash` 模式适合离线验证多轮逻辑
- `faiss` 模式复用真实向量库，效果更接近正式问答

### 12.6 在线 API
